When to Commit

Key Points

  • Commits are like snapshots of your code.
  • Best Practice: Commit every time you make a meaningful change in the code, thus creating a timeline of your progress and showing that your code didn't just appear out of nowhere.
  • Meaningful changes include things like:
    • Getting a piece of code to function the way you want it to.
    • Fixing a typo.
    • Fixing a bug.
    When you get more experience as a developer, you'll get a better feel of what should be committed.
  • When you FINALLY get something right, for example, would be a good time to commit. This way, if something breaks after a few minutes or after a few days, you can always revert back to an earlier commit (i.e. snapshot of your code) when everything was still working fine so you can see what your code looked like at that point in time.

The Seven Rules of a Great Commit Message

Well-crafted Git commit messages are the best way to communicate context about a change to fellow developers (and to your future self). A diff can tell what changed, but only the commit message can properly tell you why.

Commit messages can show whether or not a developer is a good collaborator.

  1. Separate subject from body with a blank line (for multi-line commit messages).

    Big/complicated changes will call for multi-line commit message. Use git commit to open a text editor where you can put in a multi-line commit message. The first line (subject line) will be treated as the commit message most easily visible/accessible, and collaborators will be able to view the details in the body (separated from subject with a blank line) with commands like git show or git diff or git log -p.

  2. Limit the subject line to 50 characters.
  3. Capitalize the first letter of the subejct line
  4. Do not end subject lines with periods.
  5. Use the Imperative mood in the subject line.

    Imperative mood just means 'spoken or written as if giving a command or instruction. For example:

    • Clean your room
    • Add this feature
    • Fix that bug

    This is going to be tricky at first, since we're used to the indicative mood which is all about reporting facts. For example:

    • >Fixed bug with U
    • Changing behavior of X

    As a general rule of thumb, use your commit message to complete the following sentence:

    • If applied, this commit will your subject line here.

  6. Wrap body at 72 characters
  7. Use the body to explain what and why instead of how
